# Announce Relay

LAN-facing front door to the afterwords synthesis service. Exposes two endpoints (`/synthesize` and `/health`) that bind on `0.0.0.0:7862`. The relay forwards synthesis requests to afterwords (running locally on the M5), manages a cache of pre-synthesized audio (TTL configurable), and enforces rate limits and max-text-size bounds. Requires the `data` voice from afterwords.

## Endpoints

`POST /synthesize` accepts `text` and `voice` (fixed to `data`), returns `{url, content_type, cache_age_s}` or `{error}`. Rate limited to 30 req/min per client IP; max text 600 bytes; cache valid 7 days. `GET /health` returns operational status (memory, cache size, uptime).

## Voice Contract

The relay only supports the `data` voice (configured in `RELAY_ALLOWED_VOICES`). Requests specifying any other voice are rejected. The `data` voice is generated by afterwords (synthesis runs on M5, turnaround ~2–4s per request).

## IP-Only (No mDNS)

The relay binds `0.0.0.0:7862` and RELAY_PUBLIC_BASE_URL must use **IPv4 dotted-quad** (e.g. `http://192.168.1.171:7862`), never hostname or mDNS — this ensures Raspberry Pi and other clients on the LAN can resolve and cache the URL reliably without mdns-sd lookups.

## Tests

Run from the relay root: `python -m pytest`. Tests use a temporary data dir and mock afterwords via `httpx` mock client. Requires `pytest` and `httpx` in requirements.

## Deploy

1. Copy `.env.example` to `.env` and edit (set token, afterwords URL, public URL, data dir).
2. Run `./install.sh` (creates venv, installs, validates config, registers launchd plist).
3. Service restarts automatically on reboot (launchd `RunAtLoad: true`). Logs in `relay.log` and `relay.err`.
