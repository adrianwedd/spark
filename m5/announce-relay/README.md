# Announce Relay

LAN-facing front door to the afterwords synthesis service. Binds `0.0.0.0:7862` and exposes three endpoints — `POST /announce`, `GET /audio/{name}.wav`, and `GET /health` (documented below). The relay forwards synthesis requests to afterwords (which it reaches at its localhost-only `/synthesize`, never exposed on the LAN), manages a cache of pre-synthesized audio (TTL configurable), and enforces rate limits and max-text-size bounds. Requires the `data` voice from afterwords.

## Endpoints

`POST /announce` (bearer-auth required) — body `{text, voice="data", cache=true}`. Returns `{audio_url, voice, cached, duration_s}`. Rate limited per token; max text configurable (default 600 bytes); synthesizes via afterwords and caches the WAV.

`GET /audio/{name}.wav` — unauthenticated (Chromecast cannot send auth headers). Path-traversal-guarded (UUID/hex filename validation). Serves cached WAV files from both the public cache dir and the private (cache=false) dir.

`GET /health` — unauthenticated. Returns `{status, afterwords, cache_files}` where `afterwords` is the result of a ping to the upstream synthesis service.

## Voice Contract

The relay only supports the `data` voice (configured in `RELAY_ALLOWED_VOICES`). Requests specifying any other voice are rejected. The `data` voice is generated by afterwords (synthesis runs on M5, turnaround ~2–4s per request).

## IP-Only (No mDNS)

The relay binds `0.0.0.0:7862` and RELAY_PUBLIC_BASE_URL must use **IPv4 dotted-quad** (e.g. `http://192.168.1.171:7862`), never hostname or mDNS — this ensures Raspberry Pi and other clients on the LAN can resolve and cache the URL reliably without mdns-sd lookups.

## Tests

Run from the relay root: `python -m pytest`. Tests use a temporary data dir and mock afterwords via `httpx` mock client. Requires `pytest` and `httpx` in requirements.

## Deploy

1. Copy `.env.example` to `.env` and edit (set token, afterwords URL, public URL, data dir).
2. Run `./install.sh` (creates venv, installs, validates config, registers launchd plist).
3. Service restarts automatically on reboot (launchd `RunAtLoad: true`). Logs in `relay.log` and `relay.err`.
